## 构建dockerfile文件

### 一、学习内容

Dockerfile 主要用于定义镜像的构建过程，例如指定基础镜像、复制项目文件、安装依赖、暴露端口以及设置容器启动命令；docker-compose.yml 主要用于对一个或多个容器进行编排和统一管理。

本次实验分别尝试了基于 Python Flask 应用构建镜像，以及基于 Nginx 构建静态页面镜像，并进一步结合 docker compose 使用 `build` 参数完成镜像构建与容器启动。


### 二、操作过程

#### 1. 编写 Python Flask 项目文件

首先创建了一个简单的 Flask 应用，用于测试 Dockerfile 构建镜像的完整流程。

项目目录中主要包含以下文件：

```text
dockerfile/
├── app.py
├── requirements.txt
└── dockerfile
````

其中 `requirements.txt` 用于记录 Python 项目依赖：

```text
Flask==3.0.3
```

Dockerfile 文件内容大致如下：

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .

RUN pip install -r requirements.txt

COPY . .

EXPOSE 50001

CMD ["python", "app.py"]
```

该 Dockerfile 的主要含义如下：

```text
FROM：指定基础镜像
WORKDIR：设置容器内工作目录
COPY：复制依赖文件和项目代码
RUN：在构建镜像时安装依赖
EXPOSE：声明容器内部服务端口
CMD：设置容器启动时执行的命令
```

随后执行镜像构建命令：

```bash
docker build -t myflask:1.0 .
```

构建完成后，使用以下命令启动容器：

```bash
docker run -d --name myflask -p 50001:50001 myflask:1.0
```

最后通过 curl 测试服务是否正常：

```bash
curl 127.0.0.1:50001
```

---

#### 2. 编写 Nginx 静态页面镜像

完成 Flask 镜像构建后，又进行了 Nginx 静态页面镜像构建实验。

项目目录中主要包含：

```text
lab/
├── index.html
├── dockerfile
└── docker-compose.yml
```

最初 Dockerfile 中将文件复制到了 `/app` 目录：

```dockerfile
FROM nginx:latest

WORKDIR /app

COPY . .

EXPOSE 8001
```

后续经过排查发现，Nginx 默认网页目录并不是 `/app`，而是：

```text
/usr/share/nginx/html
```

因此 Dockerfile 修改为：

```dockerfile
FROM nginx:latest

COPY index.html /usr/share/nginx/html/index.html

EXPOSE 80
```

这样构建出的镜像启动后，Nginx 才能正确读取自定义的 `index.html` 页面。

---

#### 3. 结合 Docker Compose 构建镜像

在 docker-compose.yml 中使用 `build` 参数，让 compose 自动根据当前目录下的 Dockerfile 构建镜像。

示例配置如下：

```yaml
services:
  index:
    build: .
    image: index_html:1.0
    container_name: index_html
    ports:
      - "3699:80"
```

其中：

```text
build: .              表示使用当前目录下的 Dockerfile 构建镜像
image: index_html:1.0 表示构建后的镜像名称
container_name        指定容器名称
ports                 设置宿主机端口到容器端口的映射
```

执行以下命令完成构建并启动：

```bash
docker compose up -d --build
```

查看容器状态：

```bash
docker ps
```

访问测试：

```bash
curl 127.0.0.1:3699
```

### 三、遇到的问题及解决过程

#### 问题一：pip 安装依赖超时

在构建 Flask 镜像时，执行到以下步骤失败：

```dockerfile
RUN pip install -r requirements.txt
```

报错中出现了类似内容：

```text
ReadTimeoutError
files.pythonhosted.org read timed out
```

经过分析，问题不是 Dockerfile 语法错误，而是 pip 在容器构建过程中下载依赖时网络超时。同时，`requirements.txt` 中如果只写 `flask`，pip 会自动解析多个 Flask 版本和相关依赖版本，导致解析时间变长。

解决方法是将依赖版本固定：

```text
Flask==3.0.3
```

必要时也可以在 Dockerfile 中使用国内 pip 源：

```dockerfile
RUN pip install --no-cache-dir -r requirements.txt \
    -i https://pypi.tuna.tsinghua.edu.cn/simple \
    --timeout 100
```

通过固定版本和优化下载源，可以减少依赖解析时间，提高构建成功率。

---

#### 问题二：Flask 容器启动后访问失败

构建并运行 Flask 容器后，执行：

```bash
curl 127.0.0.1:50001
```

出现连接失败。

排查后发现，容器虽然创建成功，但应用代码 `app.py` 存在语法和缩进问题，例如：

```python
m flask import Flask
```

应改为：

```python
from flask import Flask
```

同时，`if __name__ == "__main__":` 的缩进也需要放在函数外部。

正确代码结构如下：

```python
from flask import Flask

app = Flask(__name__)

@app.route("/")
def index():
    return "hello dockerfile"

if __name__ == "__main__":
    app.run(host="0.0.0.0", port=50001)
```

修改后重新构建镜像并启动容器：

```bash
docker rm -f myflask
docker build -t myflask:1.0 .
docker run -d --name myflask -p 50001:50001 myflask:1.0
```

再次访问后服务正常。

通过这个问题认识到，`docker run` 返回容器 ID 只代表容器被创建并尝试启动，并不代表应用一定正常运行。遇到访问失败时，应优先执行：

```bash
docker ps -a
docker logs 容器名
```

查看容器是否退出以及具体错误日志。

---

#### 问题三：Nginx 容器端口映射错误

在 Nginx 实验中，最初使用了类似端口映射：

```bash
-p 8001:8001
```

或者在 compose 中写成：

```yaml
ports:
  - "3699:3699"
```

访问时出现：

```text
Connection reset by peer
```

或：

```text
Empty reply from server
```

经过排查发现，Nginx 官方镜像默认监听的是容器内的 80 端口，而不是 8001 或 3699。

因此正确的端口映射应该是：

```text
宿主机端口:容器端口
```

例如：

```yaml
ports:
  - "3699:80"
```

表示访问宿主机的 3699 端口时，请求会被转发到容器内部的 80 端口。

这个问题说明，`EXPOSE` 只是声明容器内部服务可能使用的端口，并不会真正修改应用的监听端口。真正决定访问关系的是 `docker run -p` 或 compose 中的 `ports`。

---

#### 问题四：Nginx 页面文件复制路径错误

最初 Dockerfile 中使用：

```dockerfile
WORKDIR /app

COPY . .
```

该写法只是将当前目录文件复制到了容器内的 `/app` 目录，但 Nginx 默认并不会从 `/app` 目录读取网页文件。

Nginx 默认网页目录是：

```text
/usr/share/nginx/html
```

因此需要将 `index.html` 复制到该目录：

```dockerfile
COPY index.html /usr/share/nginx/html/index.html
```

修改后重新构建并启动容器，页面访问正常。

### 四、学习收获

通过本次实验，初步掌握了 Dockerfile 的基本编写方式，理解了 Dockerfile 中常用指令的作用，包括：

```text
FROM
WORKDIR
COPY
RUN
EXPOSE
CMD
```

同时也进一步理解了 Dockerfile 与 docker-compose.yml 的区别：

```text
Dockerfile 用于构建镜像
docker-compose.yml 用于运行和编排容器
```

本次实验中还重点加深了对端口映射的理解：

```text
-p 3699:80
```

表示：

```text
宿主机 3699 端口 → 容器 80 端口
```

不能简单地认为宿主机端口和容器端口必须一致，也不能把 `EXPOSE` 当成修改服务监听端口的配置。

此外，通过多次排查容器启动失败、服务访问失败、依赖安装失败等问题，也熟悉了 Docker 常用排错命令：

```bash
docker ps
docker ps -a
docker logs 容器名
docker images
docker build -t
docker run
docker compose up -d --build
docker compose down
```


## 远程应用Web代填测试

### 测试环境

- JumpServer Core：`10.1.12.78`
- 应用发布机：Windows Server `10.1.14.77`
- 测试 Web 服务：`10.1.14.251:8000`
- 测试方式：Chrome 远程应用 + Web 脚本代填

### 实施流程

1. 在 JumpServer 中创建应用发布机，Core 地址填写 `10.1.12.78` 的可访问地址。
2. 对 `10.1.14.77` 执行发布机初始化部署。
3. 仅部署 Chrome 远程应用。
4. 在 `10.1.14.251` 部署简单登录页。
5. 在 JumpServer 创建 Web 资产，地址为 `http://10.1.14.251:8000/login`。
6. 配置 Web 资产账号和脚本代填。
7. 授权测试用户访问 Web 资产。
8. 用户从工作台连接 Web 资产，验证自动登录成功。

### 遇到的问题

1. 公司 JumpServer Core 地址访问失败  
   `10.1.14.77` 访问 `https://democenter.fit2cloud.cn:16666` 时出现 SSL/TLS 信任问题，说明发布机不信任该 HTTPS 证书。

2. 发布机 Python 安装失败  
   部署时报错 `vcruntime140.dll` 被占用。排查发现是 `msedge` 进程占用，结束 Edge 进程后清理 Python 目录并重新部署。

3. Applet 同步失败  
   初始化部署时报错 `Applet not found in path: /opt/jumpserver/data/media/applets/imstsc`。原因是 Core 上存在 applet 记录但文件缺失。后续改为先“仅初始化部署”，再依次部署所需要的应用。

4. 测试 Web 服务访问异常  
   端口连通但 HTTP 请求被意外关闭，原因是 Python 测试页面模板中 CSS `{}` 与 `.format()` 冲突。修正为 `{{}}` 后恢复正常。

### 代填脚本

```json
[
  {
    "step": 1,
    "value": "{USERNAME}",
    "target": "id=username",
    "command": "type"
  },
  {
    "step": 2,
    "value": "{SECRET}",
    "target": "id=password",
    "command": "type"
  },
  {
    "step": 3,
    "value": "",
    "target": "id=login-btn",
    "command": "click"
  }
]
```

### 结果

Chrome 远程应用部署成功后，Web 资产通过脚本代填可以正常输入账号密码并自动登录。核心前提是：发布机能访问 Core，发布机能访问目标 Web 资产，Web 资产账号授权正确，元素选择器配置准确。
